# Room Android 客户端逻辑设计

本文描述 Android 客户端第一版的页面流转、状态流转、接口调用和 WebSocket 事件处理逻辑。

## 总体页面流转

```mermaid
flowchart TD
    A[启动 App] --> B{本地是否有 token}
    B -- 否 --> C[登录页]
    B -- 是 --> D[请求 /users/me]
    D -- 成功 --> E{current_room_id 是否为空}
    D -- 失败 --> C
    E -- 空 --> F[房间列表页]
    E -- 不为空 --> G[房间详情页]

    C --> H[注册页]
    C --> I[找回密码页]
    C -- 登录成功 --> D
    H -- 注册成功 --> D
    I -- 重置成功 --> C

    F -- 创建房间成功 --> G
    F -- 进入房间成功 --> G
    G -- 主动离开/断开 --> F
```

## 登录态恢复

```mermaid
sequenceDiagram
    participant App
    participant TokenStore as DataStore TokenStore
    participant Api as ApiService
    participant Nav as Navigation

    App->>TokenStore: 读取 token
    alt token 为空
        App->>Nav: 跳转登录页
    else token 存在
        App->>Api: GET /api/v1/users/me
        alt 成功
            Api-->>App: user(current_room_id)
            alt current_room_id 为空
                App->>Nav: 跳转房间列表
            else current_room_id 不为空
                App->>Nav: 跳转房间详情
            end
        else 失败
            App->>TokenStore: 清空 token
            App->>Nav: 跳转登录页
        end
    end
```

```mermaid
flowchart TD
```

## 房间列表逻辑

```mermaid
flowchart TD
    A[进入房间列表页] --> B[加载第一页房间]
    B --> C[展示房间列表]
    C --> D{用户操作}
    D -- 下拉刷新 --> E[page=1 重新加载]
    D -- 上拉到底 --> F{hasMore && !loadingMore}
    F -- 是 --> G[page+1 加载更多]
    F -- 否 --> C
    D -- 创建房间 --> H[POST /rooms]
    D -- 进入房间 --> I[POST /rooms/:id/join]
    H -- 成功 --> J[进入房间详情]
    I -- 成功 --> J
```

## 消息加载和发送

```mermaid
flowchart TD
    A[进入房间详情] --> B[加载最新消息 limit=20]
    B --> C[展示消息列表]
    C --> D{用户操作}
    D -- 上拉到顶部 --> E{hasMoreMessages && !loading}
    E -- 是 --> F[before_id=最老消息id 加载更早消息]
    F --> G[插入到消息列表头部]
    D -- 输入并发送 --> H[POST /rooms/:id/messages]
    H --> I{发送成功}
    I -- 否 --> J[展示错误]
    I -- 是 --> K[等待 WebSocket message.created]
    K --> L[追加消息]
```

说明：

- 第一版不做客户端消息 ID。
- 发送成功后不立即本地插入，优先等待后端 WebSocket 广播，避免重复消息。
- 如果后续需要更强的发送体验，可改为“本地临时消息 + WebSocket 确认”。

## WebSocket 事件处理

```mermaid
flowchart TD
    A[RoomSocket 收到事件] --> B{event.type}
    B -- member.joined --> C[追加/更新成员]
    B -- member.left --> D[移除成员并更新 current_members]
    B -- room.owner_changed --> E[更新 room.owner_id 和成员 is_owner]
    B -- message.created --> F[追加消息]
    B -- member.mic_updated --> G[更新成员 mic_status]
    B -- 其他 --> H[忽略或记录 debug 日志]
```

事件处理规则：

- `member.joined`：如果成员已存在则覆盖，否则追加。
- `member.left`：按 `user_id` 删除成员。
- `room.owner_changed`：以事件里的 `owner_id` 更新房间房主；如果事件携带成员列表，则直接覆盖成员列表。
- `message.created`：按消息 `id` 去重后追加。
- `member.mic_updated`：按 `user_id` 更新麦克风状态。

## 离开房间和断开连接

```mermaid
flowchart TD
    A[用户点击离开房间] --> B[调用 POST /rooms/:id/leave]
    B --> C{成功}
    C -- 否 --> D[展示错误]
    C -- 是 --> E[关闭 WebSocket]
    E --> F[清空当前房间状态]
    F --> G[返回房间列表]

    H[WebSocket 异常断开] --> I[按后端语义视为离开房间]
    I --> F
```

说明：

- 第一版不做断线自动保留席位。
- 页面退出房间时主动关闭 WebSocket。
- 如果 App 进入后台导致连接断开，重新进入时通过 `/users/me` 或房间接口同步真实状态。

## 错误处理

```mermaid
flowchart TD
    A[接口返回] --> B{HTTP 是否成功}
    B -- 否 --> C[网络/服务错误]
    B -- 是 --> D{code == 200}
    D -- 是 --> E[返回 data]
    D -- 否 --> F[业务错误 message]
    C --> G[ViewModel 写入 errorMessage]
    F --> G
    G --> H[页面展示 Snackbar 或错误态]
```

原则：

- 业务失败优先展示后端 `message`。
- 网络失败展示通用网络错误。
- 表单错误留在当前页面，不跳转。
- 登录态失效时清空 token 并回登录页。
