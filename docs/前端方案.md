# Room Android App 方案

## 目标

Room 第一版客户端采用原生 Android 实现，移动端优先，覆盖注册、登录、找回密码、头像上传、房间列表、创建/进入房间、房间详情、文字消息、麦克风状态和 WebSocket 房间事件同步。

第一版目标是把主链路做稳，不追求复杂社交能力、离线消息、多端状态同步或房间内复杂音视频能力。

## 技术选型

- 平台：Android 原生
- 语言：Kotlin
- UI：Jetpack Compose
- 架构：Compose + ViewModel + StateFlow
- 异步：Kotlin Coroutines + Flow
- 依赖注入：Hilt
- 网络：Retrofit + OkHttp
- JSON：kotlinx.serialization
- 图片加载：Coil
- 本地存储：DataStore，保存 token 和少量用户状态
- WebSocket：OkHttp WebSocket
- 导航：Navigation Compose

## 目录规划

```txt
room-app

    前端方案.md
  app
    src/main/java/.../room
      data
        api
        model
        repository
        websocket
        storage
      domain
      ui
        auth
        profile
        roomlist
        roomdetail
        components
        theme
      core
        network
        result
        error
```

建议边界：

- `data/api` 只放 Retrofit API 定义。
- `data/repository` 负责组合 API、DataStore、WebSocket，不写 UI 状态。
- `data/websocket` 封装房间 WebSocket 连接、事件解析和关闭。
- `di` 放 Hilt modules，提供 Retrofit、OkHttp、ApiService、DataStore、TokenStore、RoomSocket 等依赖。
- `ui/*` 使用 Hilt ViewModel 暴露 `StateFlow` 给 Compose 页面。
- 第一版不强制 Repository 层，ViewModel 可以直接注入 `ApiService`、`TokenStore`、`RoomSocket`。
- `core/network` 统一处理 token、后端业务响应结构和错误转换。

## API 对接

后端统一响应：

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

Android 端约定：

- HTTP 状态码不是主要判断依据，业务成功以 `code == 200` 为准。
- OkHttp Interceptor 自动追加 `Authorization: Bearer token`。
- token 使用 DataStore 持久化，启动时恢复登录态。
- ViewModel 或轻量错误转换工具将后端业务错误转换为统一 `AppError`，页面决定展示方式。
- 文件上传使用 `multipart/form-data`。

## WebSocket 约定

连接地址：

```txt
/api/v1/ws/rooms/:room_id
```

请求头：

```txt
Authorization: Bearer jwt_token
```

第一版事件：

- `member.joined`
- `member.left`
- `room.owner_changed`
- `message.created`
- `member.mic_updated`

Android 端处理原则：

- 进入房间成功后再建立 WebSocket。
- WebSocket 断开按后端语义等价于离开房间。
- 收到 `message.created` 后追加消息。
- 收到成员事件后更新成员列表和 `current_members`。
- 收到 `room.owner_changed` 后更新 `room.owner_id`，成员 `is_owner` 以后端返回为准。
- 页面退出房间时主动关闭 WebSocket，并调用离开房间接口。

## 状态模型

ViewModel 通过 Hilt 注入 API、TokenStore 和 RoomSocket。建议拆分：

- `AuthViewModel`：登录、注册、找回密码、token 状态
- `ProfileViewModel`：当前用户资料、更新昵称、更新头像
- `RoomListViewModel`：房间列表、分页、创建房间、进入房间
- `RoomDetailViewModel`：房间详情、成员、消息、麦克风、WebSocket 事件

关键约束：

- 当前用户同一时间只能在一个房间。
- 房主来源以后端 `room.owner_id` 为准。
- 成员对象里的 `is_owner` 是后端返回的派生字段，客户端只用于展示。
- 断开 WebSocket 后需要回到房间列表或重新同步用户当前房间状态。

## UI 方向

第一版 UI 追求清晰、轻量、适合移动端单手操作：

- 房间列表突出“当前可进入房间”和人数状态。
- 房间详情以成员状态和消息输入为核心。
- 房主标识、麦克风状态、成员进出事件需要清晰可见。
- 错误提示使用 Snackbar 或页面级错误态，不直接堆 Toast。
- 加载态、空态、错误态必须覆盖主页面。

## 第一版不做

- 不做多房间同时在线。
- 不做断线后自动保留房间席位。
- 不做消息客户端幂等 ID。
- 不做离线消息推送。
- 不做复杂音视频能力。
- 不做前端全局权限系统，只处理登录态。

## 开发顺序

1. 初始化 Android 项目、Gradle、Compose、基础 theme。
2. 封装 Retrofit、OkHttp Interceptor、DataStore token 存储。
3. 配置 Hilt modules，定义 API model 和统一响应结构。
4. 完成注册、登录、找回密码、头像上传。
5. 完成房间列表、创建房间、进入房间。
6. 完成房间详情、成员列表、历史消息、发送消息。
7. 接入 OkHttp WebSocket 事件。
8. 做真机调试、弱网测试和后端联调。

## 联调依赖

- 后端服务地址，例如 `http://127.0.0.1:9999`。
- Android 模拟器访问宿主机本地服务时通常使用 `http://10.0.2.2:9999`。
- 后端数据库已执行最新 migration。
- WebSocket 请求需要携带 `Authorization` header。
- 上传头像依赖腾讯云 COS 配置可用。
